# Training & Fine-Tune Methods to Improve Whisper ASR for Daiya

Research report for Daiya-RMT (near-real-time Thai-English / Japanese-English code-switching ASR).
Focus: how to make the next LoRA iteration of `whisper-large-v3` more accurate, more
context-aware, and better at short utterances and embedded English technical terms, without
just re-discovering the M2 recipe.

Date: 2026-07-07. All non-obvious claims are cited inline; hyperparameters not stated in a
source are marked "not stated". Evidence vs. speculation is flagged per idea.

---

## 0. Daiya's current recipe (baseline being improved)

From `docs/m2-asr-model-state.md` and `training/whisper/src/daiya_whisper_lora/train.py`:

- QLoRA (4-bit NF4, double-quant, fp16 compute) on `openai/whisper-large-v3`.
- LoRA `r=16`, `alpha=32`, `dropout=0.05`, targets `q_proj,k_proj,v_proj,out_proj,fc1,fc2`
  (attention + FFN, both encoder and decoder — PEFT wraps every module whose name matches).
- LR `2e-5`, 2 epochs, warmup 50, batch 2 x grad-accum 8 (effective 16), grad checkpointing
  (`use_reentrant=False`).
- Labels: clean-read (disfluencies removed), generated by `gemini-3.1-flash-lite` with
  previous-chunk text + rolling context. **Context columns are NOT injected as decoder prompt
  tokens during training** (`prepare_example` just does `tokenizer(text)`; `train.py:284-293`).
- Per-example language prefix token set from metadata (`set_prefix_tokens`), so Thai rows train
  `<|th|>`, English rows `<|en|>`.
- Model selection: `load_best_model_at_end` with `metric_for_best_model="eval_loss"` because
  `predict_with_generate=False` (`train.py:361`). **A WER-based generation path already exists in
  code but is disabled** (`train.py:179-189`).
- No SpecAugment, no speed perturbation, no noise augmentation in `prepare_dataset`.
- Merge → CT2 `int8_float16`, with a small WER-drift sanity check (`merge.py`).

Observed problems: slower than iter4, CER not clearly better (22.9% vs 21.2%), short utterances
fragile, embedded English terms unstable (`relation กัน` → `รีเอชันกัน`), and eval-loss picked a
worse checkpoint than generation probes (ckpt-400).

## 1. LoRA vs QLoRA vs full fine-tune — what params people actually use

### Evidence on hyperparameters

- **Official HF PEFT Whisper recipe** (`peft/examples/int8_training/peft_bnb_whisper_large_v2_training.ipynb`,
  and the "Fine-Tune Whisper with PEFT" blog): `r=32, lora_alpha=64, target_modules=["q_proj","v_proj"]`,
  `lora_dropout=0.05`, `bias="none"`, INT8 base, LR `1e-3` (much higher than Daiya's 2e-5 because
  only 2 projections train), 3 epochs. This is the canonical low-cost recipe and it only touches
  **q_proj + v_proj**. [1][2]
- **Community large fine-tunes** commonly widen targets to the full set Daiya already uses
  (`q,k,v,out_proj,fc1,fc2`) with `r=32, alpha=64` and LR `~8e-5` cosine, bf16. [3]
- **QLoRA original** (Dettmers et al.): 4-bit NF4 + double quant matches 16-bit LoRA and 16-bit
  full-FT on their benchmarks; the point of QLoRA is memory, not accuracy. [4]

### Evidence on which target modules matter

- The HF default (**q_proj, v_proj only**) already recovers most of the gain for many languages;
  adding `k_proj/out_proj/fc1/fc2` increases trainable params and can help capacity-limited
  adaptation but is not always worth it. No source cleanly isolates the marginal value of `fc1/fc2`
  for Whisper — treat "FFN targets help" as **plausible but unproven** for Daiya's data size. [1][2]
- Community discussion on decoder-only vs full: fine-tuning the **decoder** is where lexical /
  code-switch / term behavior lives; the encoder mostly needs adaptation for acoustic domain shift.
  [5]

### LoRA vs QLoRA vs full — accuracy

- A direct **Whisper** LoRA-vs-full comparison (Swiss German, subtitle-aligned) found **full
  fine-tuning superior**: best LoRA 26.28% WER vs full 25.60% WER (~0.68 pt), with better stability
  and robustness. So on medium/large data, full-FT can beat LoRA on Whisper. [6]
- But on the **same** Whisper data, properly-configured QLoRA ≈ full FT (cWER 13.8–13.9%). The gap
  only appears when LoRA is under-configured (too-low rank, too-few targets, wrong LR). [6]
- General finding: QLoRA's 4-bit sometimes acts as mild **regularization** (a study saw QLoRA
  slightly > LoRA). So Daiya's 4-bit load is unlikely to be the accuracy bottleneck. [4][7]

### Relevance / risk / experiment for Daiya

- **Relevance**: Daiya's `r=16, alpha=32, LR 2e-5` is *conservative* vs. community norms. The very
  low LR + 2 epochs + broad targets may be **under-fitting the adapter while over-fitting late
  checkpoints** (consistent with the ckpt-400 finding).
- **Risk**: raising LR/rank without a generation gate can worsen hallucination on short clips.
- **Experiment A (rank/LR sweep)**: `r∈{16,32}`, `alpha=2·r`, LR `∈{2e-5, 8e-5, 1e-4}`, 2 epochs,
  gate every run by generation-CER on a held-out + short-utterance + term probe set (see §5).
  Map: change `TrainingConfig` defaults in `train.py:76-78,63`.
- **Experiment B (target-module ablation)**: `q,v` only vs full set, same LR — quantify whether
  `k,out,fc1,fc2` actually buy CER on Daiya's ~hours-scale data. Map: `--lora-target-modules`.

## 2. Prompt-/context-conditioned Whisper training (the biggest untapped lever)

This is the single most relevant thread for Daiya, because Daiya already *generates* context
columns but never teaches the model to use `initial_prompt`. Whisper's decoder prompt mechanism
is: `<|startofprev|> <prompt tokens> <|startoftranscript|> <|lang|> <|transcribe|> …`. At inference,
`faster-whisper`'s `initial_prompt` fills the `<|startofprev|>` slot. If the model was **only**
pretrained to treat that slot as *previous transcript*, feeding it term lists behaves like a raw
GPT prompt and can hurt. Fine-tuning fixes this.

### Prompt-conditioning fine-tune (domain text) — strong evidence

- **"Zero-shot Domain-sensitive Speech Recognition with Prompt-conditioning Fine-tuning"**
  (clairaudience, MTK Research): fine-tune Whisper conditioning generation on a **domain text
  prompt** in the `<|startofprev|>` slot. Up to **33% WER reduction** on unseen domains (medical,
  air-traffic, financial). A **text-only** variant (adapt with text prompts, no new audio) still
  gave **29%** WER reduction on medical. Code/weights public (`mtkresearch/clairaudience`).
  Hyperparameters beyond "fine-tune Whisper from demonstrations with prompt examples" are **not
  stated** in the abstract/method summary. [8]
- Directly transferable: Daiya's rolling-context + Terms prompt is exactly a "domain prompt". The
  runtime already builds a per-utterance `initial_prompt` (§ M2 doc), so **train the decoder to
  honor it**.

### Hotword / rare-term biasing fine-tune — strong quantitative evidence

- **B-Whisper** ("Improving Rare-Word Recognition of Whisper in Zero-Shot Settings"): supervised
  fine-tune where each training example gets a **bias-word list** (1 true-bias word + 25–150
  false-bias distractors, shuffled) placed as prompt tokens; weighted cross-entropy (β=1.1 on
  true-bias tokens); randomly drop the true word (p=0.3) or whole list (p=0.2) for robustness.
  LR `1e-7` linear decay, dropout 10%, **1 epoch**, 670h Common Voice English. Results:
  Rare-WER 18.0→11.2% (−45.6% rel), OOV-WER 37.1→16.0% (−60.8% rel), overall WER 12.1→8.9%, and it
  **generalized to unseen languages** (FR/ES/IT/DE OOV) despite English-only training. Crucially:
  **un-fine-tuned Whisper + bias prompt got worse** ("Whisper's prompt operates like a base GPT
  model") — proving the prompt slot must be *trained*, not just used. [9]
- **Multitask contextual-biasing + keyword spotting** and **Dolphin-CN-Dialect** use the same
  pattern: dynamically sample a hotword sublist per utterance, insert between special prompt tokens,
  train decoder to copy them. (PDFs not text-extractable here, but the abstracts confirm the
  recipe.) [10][11]

### Relevance / risk / experiment for Daiya

- **Relevance (very high)**: this attacks two of Daiya's five documented flaws at once — English
  technical terms and context awareness — and can also help short utterances (a short clip plus a
  strong prior is easier to decode).
- **Risk**: if you *only* ever train with a correct term present, the model learns to hallucinate
  the prompt into the output ("prompt leakage"). B-Whisper's distractor words + drop probabilities
  are the antidote; copy them. Also keep prompt token length bounded (Whisper prompt budget is
  ~half the 448 context).
- **Experiment C (prompt-conditioned training)** — the flagship next run:
  1. In `training/processor/whisper` labeler, you already store `context_before`, `context_after`,
     `notes`, English terms. Build a per-row prompt = rolling previous text tail + a shuffled term
     list (true terms from this row + distractor terms sampled from the corpus's term pool).
  2. In `train.py:prepare_example`, tokenize with the prompt in the `<|startofprev|>` slot:
     use `processor.tokenizer.get_prompt_ids(prompt)` (or manually prepend `<|startofprev|>`),
     concatenate `[prompt_ids] + [sot… label_ids]`, and **mask the prompt tokens to -100** so loss
     is only on the transcript. This is the key code change; today `prepare_example` ignores context.
  3. Apply drop probabilities (row term p≈0.3, whole prompt p≈0.2) so the model still works with no
     prompt.
  4. Evaluate with `lab/asr_eval.py --strategy rolling_initial_prompt` (already implemented,
     `asr_eval.py:765-773`) and a new **term-probe** subset. The M2 doc already shows
     `rolling_initial_prompt` helps *without* training; training should widen that gap.

## 3. Code-switching fine-tuning of Whisper / multilingual ASR

### Language-token handling

- Daiya sets **one** language token per row (`th` or `en`). For genuinely code-switched utterances
  this is a mismatch: the reference is Thai+English but the prefix says `<|th|>`. Options in the
  literature:
  - **Concatenated / soft language embeddings**: language-specific embeddings concatenated with
    acoustic features before the encoder improves CS-ASR. [12]
  - **Soft prompt tuning for CS** (Interspeech 2025, "Adapting Whisper for Parameter-efficient
    Code-Switching ASR via Soft Prompt Tuning"): learnable soft prompts beat LoRA and approach full
    FT on **SEAME + ASCEND** with far fewer params; language tokens conditioned per-utterance.
    Exact MER numbers not text-extractable from the PDF here. [13]
  - **Prompt/language-token engineering** for SEAME/ASRU2019 matters mainly in **zero-shot**;
    after fine-tuning, the single- vs dual-token choice washes out. [14]
- Practical takeaway for Daiya: for CS rows, either (a) keep `<|th|>` as the matrix language and let
  fine-tuning absorb the English inserts (simplest, and what most CS Whisper fine-tunes do), or
  (b) experiment with leaving language **unforced** (`language=None`) so Whisper's own LID picks per
  segment. Daiya's `language_policy` already supports `none` (`train.py:318-325`).

### Datasets / corpora

- **Code-switch benchmarks**: SEAME, ASCEND (Mandarin-English) dominate the literature; no
  standard **Thai-English** or **Japanese-English** CS corpus surfaced — consistent with Daiya's
  need to build its own labeled data. [13][14]
- **Thai (non-CS) references worth mining for recipe + eval**:
  - **Thonburian Whisper** (biodatlab): large-v2/v3 fine-tuned on Common Voice + FLEURS + Gowajee +
    curated Thai; WER measured with **deepcut** tokenizer after normalization — directly relevant to
    Daiya's Thai-spacing WER unreliability problem. They also ship **distil-whisper-th**. [15]
  - **Pathumma-whisper-th-large-v3** (NECTEC) and a LoRA noise-robust variant. [15]
- **Japanese**: Kotoba-Whisper (§7) and its **bilingual** variant (`kotoba-whisper-bilingual-v1.0`)
  are the closest analog for JA-EN. [16]

### Relevance / experiment

- **Experiment D (CS language-token policy)**: 3-way compare on a CS-heavy validation slice —
  forced `<|th|>`, forced `<|en|>` on English-dominant rows, and `language_policy=none`. Metric:
  term-CER + overall CER. Cheap; only changes `--language-policy` / metadata language field.
- **Experiment E (Thai WER metric fix)**: adopt **deepcut**-tokenized WER (as Thonburian does) in
  `lab/asr_eval.py` so Thai word-boundary noise stops masking real gains. The M2 doc explicitly
  flags Thai WER as unreliable; this is a low-risk measurement upgrade. [15]

## 4. Verbatim vs clean-read labels (Daiya uses clean-read)

Daiya's label policy strips disfluencies (per MEMORY: "clean-read labels, no disfluencies"). The M2
doc even hypothesizes the CER/WER "degradation" is the model correctly dropping fillers/repeats.

### Evidence

- **The mismatch is real and measurable**: comparing a verbatim reference against a non-verbatim
  hypothesis (or vice-versa) inflates apparent WER even between two humans — an "error" that is
  actually a style difference. So Daiya's clean-read labels + any verbatim-leaning eval reference =
  artificially high CER/WER. [17]
- **CrisperWhisper**: shows the *opposite* direction — most web training data is **cleaned/fluent**,
  so stock Whisper is biased to omit disfluencies; making it verbatim required **tokenizer changes +
  fine-tuning on filler-labeled data**, and doing so reduced WER on verbatim datasets (AMI
  16.82→9.72%) **and cut hallucinations** (empty-transcript noise samples in 1% of training data;
  drop tokens <50ms to kill loops). [18][19]
- Key transferable insight: **hallucination mitigation via empty/nonsense-transcript negative
  samples** (1% of data) is cheap and directly relevant to Daiya's short-utterance failures where
  the model invents text for near-silence. [18]

### Relevance / risk / experiment

- **Relevance**: Daiya's design goal *wants* clean-read (drop fillers). That is a legitimate choice —
  but then the **eval reference must also be clean-read**, or CER is measuring style, not accuracy.
  This likely explains part of "CER not clearly better".
- **Experiment F (label-style consistency audit)**: verify the 32-sample benchmark references are
  clean-read in the *same* convention as training labels. If not, re-normalize references (drop
  fillers/repeats) before scoring in `lab/asr_eval.py`. Expect apparent CER to drop.
- **Experiment G (negative/empty samples)**: add ~1% empty-transcript noise/silence clips to teach
  the model to emit nothing on non-speech — targets short-utterance hallucination directly. [18]

## 5. Checkpoint / model selection for generative ASR (Daiya's known bug)

### Evidence: teacher-forced loss misranks checkpoints

- Widely reported for Whisper: **validation loss keeps dropping while validation WER rises**; the
  best-WER checkpoint is often **early** in fine-tuning. This is exactly Daiya's ckpt-400 vs
  best-loss finding. [20][21]
- Mechanism: cross-entropy under teacher forcing rewards confident next-token prediction on
  gold-prefix; generation (free-running, with exposure bias) is what WER measures. As the model
  over-fits, it gets more *confident* (lower CE) without transcribing better. [20]
- **Fix (consensus)**: select by **generation WER/CER**, not loss. Save many checkpoints, decode a
  held-out set, pick lowest WER; use **early stopping on WER**. [20][21]

### Additional levers

- **EMA of weights** and **best-by-WER** are standard for stabilizing ASR fine-tunes (general ASR
  practice; not Whisper-specific in the sources gathered — treat EMA as **plausible, untested here**).
- **SpecAugment** during training both regularizes (delays the loss/WER divergence) and improves
  robustness (§6). [22]

### Relevance / experiment (highest-priority fix)

- Daiya **already has the code path**: setting `predict_with_generate=True` flips
  `metric_for_best_model` to `"wer"` and wires the `compute_metrics` WER function
  (`train.py:179-189, 361`). The blocker is that HF generation-eval is slow and its WER is the same
  Thai-spacing-fragile metric.
- **Experiment H (generation gate)** — do this before any other run:
  1. Enable `predict_with_generate=True` **or** add an external eval callback that, every
     `save_steps`, runs `lab/asr_eval.py`-style CER (with the Thai-spacing normalization already in
     `asr_eval.py:239-246` and deepcut WER from §3) on a fixed held-out + short + term probe set.
  2. Select the checkpoint by that CER, not eval_loss. Add early stopping on CER patience.
  3. Concretely validate the existing ckpt-400 claim on a **larger** held-out set (M2 doc "next step
     3").
- **Overfitting signature to watch**: loss ↓ while probe-CER ↑, and short-utterance CER degrading
  faster than overall — both already visible in Daiya's logs.

## 6. Data augmentation for robustness & short utterances

### Evidence

- **SpecAugment** (time + frequency masking on the mel spectrogram) is the highest-value, cheapest
  augmentation; it's part of Whisper's own pretraining and the standard ASR fine-tune regularizer.
  [22]
- **Speed perturbation** (0.9/1.0/1.1x resample) triples data, preserves timbre/content, adds
  rate-of-speech variety; combined with SpecAugment + noise injection, fine-tuned Whisper
  "consistently outperforms" the non-augmented baseline; a HuBERT speed-perturb run cut WER >28%.
  [22]
- **Noise injection / empty-transcript negatives** improve robustness and cut hallucination (§4,
  CrisperWhisper). [18][22]
- Short-clip handling: Whisper always pads to 30s; very short clips are fragile because there's
  little acoustic evidence and the LM prior dominates → hallucination/loops. Mitigations that the
  literature + Daiya's own logs support: (a) prompt-conditioning to supply a strong prior (§2),
  (b) empty-negative training (§4), (c) the temperature-ladder / fallback tuning Daiya already did,
  (d) merging tiny VAD chunks (Daiya tested `merged_deferred_short`; it regressed at runtime, but
  **training** on slightly longer merged windows is different and untested).

### Relevance / risk / experiment

- **Relevance**: SpecAugment is currently **absent** from `prepare_dataset` — a clear, low-risk add
  that both regularizes (helps the loss/WER divergence in §5) and improves noisy real-world robustness.
- **Risk**: over-aggressive masking on already-short Thai clips can erase the whole utterance; use
  conservative masks.
- **Experiment I (SpecAugment)**: apply time/freq masking to `input_features` inside
  `prepare_example` (or via a collator wrapper in `train.py:91`) — e.g. 2 freq masks (F≈27) + 2 time
  masks (T≈40, p≈0.05), Whisper-style. Gate with the §5 generation eval.
- **Experiment J (speed perturbation)**: pre-augment the dataset x1.5–x3 with sox/torchaudio
  resample at 0.9/1.1 before feature extraction. Higher cost (bigger cache), so run after
  SpecAugment shows signal.

## 7. Distillation / smaller-faster models as a 1st-pass (2-vs-3-pass question)

Daiya's open research Q4 is 2-pass vs 3-pass. A distilled model is the natural **pass-1**.

### Options and tradeoffs (evidence)

- **Distil-Whisper**: freeze/copy the teacher **encoder**, keep a **2-layer decoder** (maximally
  spaced init), distill on large-scale **pseudo-labels filtered by WER heuristic**. Result: ~5.8–6x
  faster, ~49–51% smaller, **within ~1% WER** on out-of-distribution short-form. The decoder is
  >90% of inference time, so cutting decoder layers is where the speed comes from. English-first;
  distil-large-v3.5 is ~1.5x faster than large-v3-turbo and slightly better on short-form. [23][24]
- **whisper-large-v3-turbo**: same idea shipped by OpenAI — decoder 32→4 layers, 809M params
  (vs 1.55B). Fast, strong multilingual, a ready-made pass-1 or even pass-2 candidate. [24]
- **Kotoba-Whisper** (JA): teacher large-v3 encoder + **2-layer decoder**, trained on ReazonSpeech
  (v1: 1,253h; v2: all subset), transcripts with **>10 WER filtered out**, 8 epochs, batch 256.
  Beats large-v3 CER/WER in-domain, competitive OOD (JSUT, CommonVoice-ja). A **bilingual** variant
  exists — the closest ready model for Daiya's JA-EN path. [16][25]
- **Thonburian distil-whisper-th** (Thai): distilled Thai Whisper — closest ready pass-1 for the
  Thai path. [15]

### Relevance / risk / experiment

- **Relevance**: for 3-pass, Daiya wants a fast, *good-enough* pass-1. distil-large-v3.5 or
  large-v3-turbo (multilingual, no training) are drop-in `faster-whisper` models; Kotoba-bilingual
  and Thonburian-distil are language-specialized.
- **Risk**: distilled models hallucinate more on long-form and on languages under-represented in the
  pseudo-label set; Daiya's own **large-v3 M2 remains pass-2** so pass-1 errors are correctable.
- **Experiment K (pass-1 bake-off)**: run `lab/asr_eval.py` on the 32-sample (and a larger) set for
  `large-v3-turbo`, `distil-large-v3.5`, `kotoba-whisper-bilingual`, `thonburian distil-whisper-th`
  — record CER **and** RTF. Decision rule: if a distilled model hits ≤ ~1.5x M2 CER at ≥3x speed,
  3-pass is justified; else stay 2-pass with M2 as both fast and accurate pass.
- **Experiment L (distill Daiya's own model)**: once a strong prompt-conditioned large-v3 exists,
  distill it to a 2-layer-decoder student on Daiya audio + pseudo-labels (WER-filtered) for a fast
  pass-1 that shares Daiya's code-switch behavior. Higher effort; defer until §2 lands.

## 8. Merge + CT2 conversion pitfalls affecting accuracy

Daiya already merges LoRA → CT2 `int8_float16` (`merge.py`). Findings:

- **int8 quality loss is small**: CTranslate2 int8 typically <2% accuracy loss; on whisper-small,
  int8 changed WER by only ~0.1 pt. So `int8_float16` is **unlikely** to explain M2's CER — the
  initial bad-CER run was a decode/fallback bug, not quantization (M2 doc confirms). [26][27][29]
- **Tokenizer files must ship with the CT2 model**: Daiya's `merge.py:186-191` already copies
  `tokenizer.json` + `preprocessor_config.json` — important, because without them faster-whisper
  falls back to a default whose **large-v3 token ids are shifted** (task token becomes
  `<|translate|>`). Keep this. [26]
- **Word timestamps need `alignment_heads`**: faster-whisper computes word timestamps via
  cross-attention DTW using a preset list of **alignment heads** per model size. A **merged +
  CT2-converted custom model** may not carry the right alignment-head metadata, degrading word
  timestamps (relevant because `asr_eval.py` and the diarization multiplexer use
  `word_timestamps=True`). CrisperWhisper shows how sensitive timestamps are to which heads/tokens
  are used. Verify the CT2 model exposes alignment heads; if not, word timestamps fall back to
  segment-level. [26][28][18]
- **Merge dtype**: `merge.py` loads base in fp16 on CUDA then merges — fine, but merging LoRA in
  fp16 (vs bf16 training compute) can introduce tiny drift; the existing WER-drift sanity check
  (`allowed_absolute_wer_drift=1.0`) guards this. Consider merging in **fp32** then converting, to
  minimize merge rounding before quantization.

### Experiment M (CT2 fidelity check)

- Extend `merge.py`'s sanity check to also compare **CER** (not just WER) and **word-timestamp
  availability** between the PEFT model and the CT2 model on the probe set. Confirms quantization +
  alignment-head handling aren't silently costing accuracy or timestamps.

## 9. Prioritized experiment roadmap for Daiya

Ordered by expected value / effort:

1. **Experiment H (generation-based checkpoint gate)** — fixes the known model-selection bug; unlocks
   trustworthy comparison for everything else. Enable existing `predict_with_generate` path or an
   external CER callback. (§5)
2. **Experiment C (prompt-conditioned training)** — teach the decoder to use the rolling-context +
   term prompt via `<|startofprev|>` with distractors + drop-out. Directly targets context awareness
   and English-term instability. Strongest evidence (33% WER↓ domain, 45–60% rare-word↓). (§2)
3. **Experiment F/E (label-style + deepcut WER audit)** — make sure CER/WER measure accuracy, not
   Thai spacing or verbatim style. Cheap; may reveal M2 is already better than it looks. (§3,§4)
4. **Experiment I (SpecAugment)** — regularize + robustify; complements #1. (§6)
5. **Experiment G (empty-negative samples)** — cut short-utterance hallucination. (§4,§6)
6. **Experiment A/B (rank/LR + target-module sweep)** — the current LR 2e-5 is low vs norms;
   re-tune under the new gate. (§1)
7. **Experiment K (pass-1 distilled bake-off)** — settle 2-vs-3-pass with data. (§7)
8. **Experiment D (CS language-token policy)** and **M (CT2 fidelity)** — smaller refinements. (§3,§8)

### File map

- Adapter config / trainer / model-selection: `training/whisper/src/daiya_whisper_lora/train.py`
  (`TrainingConfig` §60-88, `prepare_example` §284-293 for prompt injection + SpecAugment,
  `build_training_args` §361 for the generation gate, `load_lora_model` §433-440 for targets).
- Label/prompt generation: `training/processor/whisper/src/daiya_whisper_pipeline/` (`llm.py`,
  `pipeline.py`) — add shuffled term-list + distractor prompt columns.
- Merge / CT2 / fidelity: `training/whisper/src/daiya_whisper_lora/merge.py`.
- Evaluation / gates / probes: `lab/asr_eval.py` (add deepcut WER, term-probe subset, generation
  gate reuse; `rolling_initial_prompt` strategy already at §765-773).
